# HR 简历解析、批量筛选与岗位匹配

## 目标

把简历解析拆成三个确定性模块：

1. **模块 A：简历解析增强**，输出可信的结构化候选人卡片。
2. **模块 B：匹配分评分引擎**，输出统一的 0-100「匹配分」（四维平滑加权）与由它单调推导的推荐等级（强推荐/待审核/谨慎/不推荐）。
3. **模块 C：异常校验引擎**，兜住履历时间线、低可信经历、岗位跳跃、频繁跳槽、学历缺失等低级误判。
4. **模块 D：批量岗位匹配**，对简历包和目标 JD 输出 HR 可复核的排序表。

默认场景是销售岗 HR 快筛；其他岗位也可使用通用解析、工龄和异常能力。

## 最重要的三条（务必先读）

1. **分数和等级一律以脚本输出为准**：匹配分（0-100）、推荐等级、稳定分、Gap 分、工龄都由脚本算好，**不要自己心算、估算或调整数字**，也不要自行改判推荐等级。
2. **批量筛选优先直接运行** `scripts/batch_screen_resumes.py`，把它输出的表格**原样**呈现给 HR（列即：排序、候选人、匹配分、推荐等级、学历、相关经验、简要分析）；不要绕过脚本自行打分排序。**简要分析就写在表格最后一列，不要在表外再单列一段分析。**
3. **单份用** `scripts/parse_resume.py` 拿到 JSON 后，只做“读字段、填模板”，不改任何数值；字段缺失就写“未说明”，宁可少写也不要编造。

## 快速使用

### 单份解析

```bash
python3 scripts/parse_resume.py path/to/resume.txt --jd path/to/jd.txt --job-title 销售
```

也可以在代码中调用：

```python
from scripts.parse_resume import parse_resume_text

candidate = parse_resume_text(resume_text, jd_text=jd_text, job_title="电话销售")
```

主入口会依次完成文本抽取、基本信息解析、经历切分、岗位标准化、经历分类、时长计算、重叠标记、可信度评分、稳定性/GAP 评分、异常校验和匹配分评分（含由匹配分推导的推荐等级）。

### 批量筛选

```bash
python3 scripts/batch_screen_resumes.py path/to/resumes --jd path/to/jd.txt --job-title 电话销售
```

批量入口会扫描目录中的 `.pdf`、`.docx`、`.txt`、`.md` 简历，逐份调用单份解析引擎，并输出推荐排序表。图片类简历（`.jpg/.jpeg/.png`）若环境装有 OCR 库（pytesseract 或 paddleocr）会自动 OCR 解析并标注“建议人工二次确认”，否则降级为“需人工处理”，不影响其他简历。

### 目标岗位与 JD 来源（重要）

筛选哪个岗位、用什么门槛，**完全由运行时这两个参数决定；脚本不会自动扫描任何目录找 JD**：

- `--job-title <岗位>`：目标岗位（如 `电话销售`），决定与候选人经历算相关度的基准。
- `--jd <JD文件路径或文本>`：这份 JD 会被解析出学历、技能关键词、最低经验年限等门槛。

各岗位 JD 由 HR 维护在 skill 目录下的 `knowledge/` 里（每个岗位一个 `.txt`，文件名=岗位名；用法与约定见 `knowledge/README.md`，已附示例 `knowledge/外贸B2B销售.txt`）。**要筛某个岗位，就在运行时把对应 JD 文件路径传给 `--jd`**：

```bash
python3 scripts/batch_screen_resumes.py path/to/resumes --jd knowledge/外贸B2B销售.txt --job-title 电销
```

> **给 agent 的约定（JD 处理流程，务必照做，不要一上来就问“JD 是什么”）**：
> 1. **先自己看 `knowledge/` 目录**：列出里面已有的 JD 文件（`ls knowledge/*.txt`），看有没有跟本次要筛岗位对应的。
> 2. **有对应的**：直接把它的**路径**作为 `--jd` 传入运行；若一眼能对上（如只有一个、或岗位名明确匹配），可直接用并在结果里注明“使用的 JD：knowledge/xxx.txt”，不必反复追问。
> 3. **有多个、拿不准用哪个**：把 `knowledge/` 里候选的 JD 列给用户，让其确认选哪一个，而不是让用户从零提供。
> 4. **`knowledge/` 里没有对应岗位的 JD**：提示用户“`knowledge/` 里暂无该岗位 JD，请把该岗位 JD 存成 `knowledge/<岗位>.txt` 再筛”，或让其直接贴 JD 文本——**不要凭记忆编造 JD、也不要不带 JD 空跑**。
> 5. 仅当确实没有 JD、也没传 `--job-title` 时，脚本才按“销售”通用口径兜底（此时不读取 `knowledge/`）。

可选参数：

- `--pass-threshold 75`：强推荐门槛（匹配分 0-100），默认 75；待审核=门槛-10、谨慎=门槛-20、更低为不推荐。也可用环境变量 `HR_PASS_THRESHOLD` 设置（优先级 CLI > 环境变量 > 默认）。单份 `parse_resume.py` 同样支持，单份与批量口径一致。
- `--output result.md|result.csv|result.json`：保存 Markdown、CSV 或 JSON；不传则打印 Markdown 表。CSV 保留全部明细列（解析置信度/命中关键词/不匹配项/人工复核原因）。
- `--weights path/to/权重.docx`：**已弃用**，仅保留兼容。评分已改为单一匹配分，权重不再参与打分。

## 输出字段

脚本会自动生成以下字段（供你读取填模板，**无需逐个核对是否齐全，也无需手动补算**）：

- `basic_info`：姓名、年龄、手机、邮箱、出生日期、`languages`（语言能力，未提及为“未说明”）。
- `education`：学校、学历、专业、起止时间。
- `experiences`：经历类型、公司、原岗位、标准岗位、起止日期、月数、可信度分、重叠标记、描述。
- `tenure_summary`：`full_time_months`、`internship_months`、`project_months`、`freelance_months`、`pending_months`、`full_time_years`。
- `stability_scores`：履历稳定分、Gap 分及标签、`gap_summary`（空窗总体说明）、`gap_score_breakdown`（Gap 分的三项构成说明）、`gap_details`（逐段空窗明细）、平均/中位在岗月数、短任职占比、最长 Gap、近 5 年 Gap。
- `resume_recency`：简历时效——`latest_year_in_text`、`years_since_latest`、`has_ongoing`、`is_stale`。
- `anomalies`：P0/P1/P2 异常、描述和动作（带“不参与评分”的项是合规/提醒类，不影响分数）。
- `recommendation`：`score_100`（0-100 统一「匹配分」，唯一底层分数）、`tier`（推荐等级：强推荐/待审核/谨慎/不推荐）、`tier_dot`（色点 🟢🟡🟠🔴）、`tier_badge`（有 P0 时为 ⚠️，否则空）、`tier_display`（色点+等级+⚠️，直接展示用）、`pass_threshold`（本次强推荐门槛）、`p0_remark`（P0 红旗文案，有 ⚠️ 时非空）、理由、置信度、目标岗位、强/弱/降权细节。（`status`/`result` 为兼容键，已置为 `tier`；`score` 为 0-1 内部分。）
- `parsing_confidence`：解析置信度（已并入匹配分 D4 维度，不再单独硬降级）。

单份卡片**可选**在末尾附一张总结表（模板见 `assets/candidate_card_template.md` 的“总结”段）；附表时所有数值必须直接取自脚本 JSON，缺失项写“未说明”。不确定时宁可只贴脚本输出的关键字段，也不要自己填数。

批量筛选表（脚本自动输出，即调用本 skill 后给 HR 的展示格式）含：**排序、候选人、匹配分（0-100）、推荐等级（强推荐/待审核/谨慎/不推荐，带色点，含 P0 时追加 ⚠️）、学历、相关经验(月)、简要分析**。**简要分析**为最后一列，一句话说清岗位背景+相关经验年限+学历+最匹配点/首要风险（带 P0 时粗体写出具体红旗）——这就是给 HR 的分析结论，**不要在表格外再单独列一段「简要分析」**。更细的解析置信度/命中关键词/不匹配项/人工复核原因用 `--output result.csv` 导出。

## 核心规则

### A. 简历解析增强

1. **岗位标准化**：用 `scripts/standardize_job_title.py` 合并同义岗位。销售相关岗位要区分电话销售、面销、大客户销售、渠道销售、商务拓展、客服等；岗位相近度用于推荐匹配。
2. **经历类型先分类再计算**：每段经历必须是正式工作、实习、校园项目、自由职业/创业或待确认。只把正式工作累计为主工龄，实习和项目单独统计。应届生在校期间的兼职/实习销售按“实习”计，不计入正式工龄。
3. **时间按月计算**：缺失日期不能默认为至今；时间倒置要进异常。多段重叠经历不得简单累加，主经历按最长正式经历优先，其他标记为 `重叠经历`。
4. **稳定性和 Gap**：销售岗输出履历稳定分和 Gap 分。脚本已附 `gap_summary`、`gap_score_breakdown`（分数构成）和逐段 `gap_details`，直接展示即可，不要只贴一个分数。
5. **解析置信度**：由关键字段完整度和低可信经历比例计算。

> 附：脚本还会自动产出 `basic_info.languages`（语言及证书）和 `resume_recency`（简历时效，最新信息距今约 2 年以上时给“可能未更新”P2 提示，仅提醒、不参与评分）——直接读字段即可，无需手动操作。

### B. 匹配分与推荐等级

评分不能只看关键词，要看证据强度。脚本输出**一个 0-100「匹配分」**（唯一底层分数），**推荐等级由匹配分单调推导**（分越高等级越高，二者永不矛盾）：

- 匹配分 = D1 经验(40) + D2 学历(15) + D3 稳定(25) + D4 置信度(20) − 降权 − P1/P2 软扣分。**四维平滑加权，缺一项只按比例扣分、不再一票归零**（修复真实资深候选人被压低分的问题）。
- 推荐等级：`🟢 强推荐` ≥ 门槛（默认 75）；`🟡 待审核` 门槛-10 起（默认 65–75）；`🟠 谨慎` 门槛-20 起；`🔴 不推荐` 更低。门槛可用 `--pass-threshold`/`HR_PASS_THRESHOLD` 调整。
- **P0 数据红旗**（时间倒置、同期多份全职、简历可能未更新等）以 `⚠️` 徽标呈现在等级旁 + `p0_remark` 加粗写出具体内容，**不扣分、不改变分档**——是否录用交给 HR 判断。
- 解析置信度已并入 D4，低置信度自然拉低匹配分，不再单独硬降级。

判据/降权仅用于生成理由文案与喂各维度（不再作硬门）：弱判据含客户沟通、简历完整、稳定性尚可、业务结果指标、可迁移；降权含低可信相关经历、相关经历久远、时间线异常、近期频繁跳槽、稳定分低、Gap 分低。详见 `references/recommendation_logic.md`。

### C. 异常校验

异常不等于淘汰，只决定是否人工复核、是否降权、是否增加追问。

- **P0 拦截类**：同期多段**非在校**全职（不同公司）、同期 3 段非在校经历、时间倒置、近期极频繁跳槽。**「一边上学一边实习/上班」是常态**——重叠里含实习/校园项目、或任一段在毕业前（在校期间）的，一律作「在校并行经历」P2 提示（不参与评分、不降权、不算不可信）；被误切成经历的教育条目（公司/岗位含院校名）不参与重叠判定。
- **P1 降权类**：低可信经历、岗位跳跃、学历层级缺失、近期跳槽偏高。
- **P2 提示类**：时间缺失、项目重叠、学校缺失、跨职能变化、轻度频繁变动、在校并行实习/兼职/项目、简历信息可能未更新（最新信息距今约 2 年以上）、候选人自述年龄与履历时间线明显不一致。带“不参与评分”标记的提示只供人工复核，不影响分数。

### D. 业务红线

- 匹配分/推荐等级只做**排序与优先级提示**，不做自动淘汰；是否推进、是否复核由 HR 决定。P0 红旗（⚠️）务必人工看过再定。
- 不将性别、年龄、籍贯、婚育、民族、照片、院校歧视等敏感属性纳入评分或排序。
- 年龄只可在候选人主动提供且与履历时间线明显矛盾时作为人工复核提示，默认不参与匹配分。
- 不主动联系候选人，不代发面试或拒绝通知。

## 验收指标

- 解析字段准确率：姓名、联系方式、教育、经历起止时间、岗位字段。
- 工龄计算正确率：正式工作、实习、项目分项月数，重叠不重复累计。
- 异常命中率：时间重叠、低可信、频繁跳槽、学历缺失、合规复核提示。
- 人工复核纠正率：P0/P1 是否真正帮助 HR 定位争议点。
- 匹配分/推荐等级的准确率与误高估率：重点监控“一行销售经历被误判为高匹配分/强推荐”。
- 批量排序可解释性：每个候选人的命中点、不匹配项、复核原因都能追溯到候选人卡片或 JD。
